# Архитектура состояния

[← Реестр протоколов](protocol_index.md) | [Загрузка контекста](context_loading.md) | [Маркер ответа](response_marker.md) | [Сервисное состояние](../State/service_state.md) | [Состояние исполнения](../State/execution_state.md)

## Назначение

Этот протокол задаёт единую модель состояния `Concept Builder`. Состояние нужно как исполнимый контракт между режимом, загруженным контекстом, разрешённой областью записи, текущей задачей, проверкой и маркером ответа.

## Источники состояния

| Файл | Режим | Назначение | Владелец |
|---|---|---|---|
| `State/service_state.md` | `Service Mode` | состояние обслуживания системы, протоколов, реестров, проверки и сервисных задач | `Service Mode` |
| `State/execution_state.md` | `Execution Mode` | состояние выбранной концепции внутри `Concepts/`, её локального реестра, результата/экспорта и связанной исполнительной задачи | `Execution Mode` |

Состояние другого режима читается как ссылочный контекст и меняется только при явной передаче, ремонтной задаче или сервисном исправлении, где целевой путь указан в контракте.

## Каноническая схема

Каждый файл состояния обязан содержать эти поля в блоке `text`.

| Поле | Обязательность | Смысл |
|---|---|---|
| `state_id` | обязательно | уникальный технический идентификатор снимка состояния |
| `owner` | обязательно | режим-владелец файла состояния |
| `active_mode` | обязательно | активный режим: `Service Mode` или `Execution Mode` |
| `active_object` | обязательно | объект работы: рабочий репозиторий, выбранная концепция или конкретная область |
| `active_issue` | обязательно | текущая задача или `none`, если задача ещё не создана |
| `status` | обязательно | рабочий статус без финального утверждения о закрытии |
| `mode_owner` | обязательно | режим, которому принадлежит текущая запись |
| `mode_boundary_status` | обязательно | проверка, что действие не смешивает области записи |
| `dialogue_state` | обязательно | где находится диалог или исполнение: `executing`, `waiting`, `blocked`, `recovery` |
| `current_focus` | обязательно | краткая формулировка текущей цели |
| `current_stage` | обязательно | технический этап текущего шага |
| `required_protocols` | обязательно | протоколы, которые должны быть загружены для действия |
| `loaded_context_scope` | обязательно | файлы, фактически загруженные для текущего шага |
| `loaded_protocols` | обязательно | протоколы, фактически прочитанные или использованные |
| `allowed_read_scope` | обязательно | допустимые области чтения |
| `allowed_write_scope` | обязательно | допустимые области записи |
| `persistence_status` | обязательно | состояние записи и перечитывания: `not_written`, `written_unverified`, `readback_required`, `partial`, `blocked`, `conflict`, `failed`. Финальные ярлыки без доказательств запрещены |
| `updated_at` | обязательно | дата обновления состояния |
| `next_step` | обязательно | следующий ограниченный шаг |
| `return_anchor` | обязательно | файл, с которого безопасно продолжать |

Дополнительные поля разрешены, если они уточняют ветку, базовый SHA, метаданные инструкции, проверку или готовность экспорта и не конфликтуют с канонической схемой.

## Связь состояния и маркера ответа

| Поле маркера ответа | Канонический источник состояния | Правило |
|---|---|---|
| `mode` | `active_mode` | значение должно совпадать дословно |
| `active_object` | `active_object` | для сервиса — рабочий репозиторий; для исполнения — выбранная концепция |
| `active_issue` | `active_issue` | не подменять `none` фиктивной задачей |
| `stage` | `current_stage` | в маркере используется короткое имя `stage`, в состоянии — каноническое `current_stage` |
| `loaded_context` | `loaded_context_scope` | не заявлять файлы, которые не были прочитаны |
| `persistence_status` | `persistence_status` | не использовать `synced`, `passed`, `Ready`, `OK` без перечитывания и доказательств |
| `next_step` | `next_step` | следующий ограниченный шаг, а не весь маршрут |
| `return_anchor` | `return_anchor` | должен указывать существующий рабочий файл |

## Правила обновления

1. Перед чтением дополнительных файлов агент читает состояние активного режима и `Protocols/protocol_index.md`.
2. Перед записью агент сверяет `allowed_write_scope` с каждым целевым путём.
3. Если действие меняет состояние, задачу, реестр или проверку, эти изменения фиксируются в одном ограниченном сегменте исправления.
4. После успешной записи агент перечитывает изменённые файлы из `GitHub` и обновляет `Validation/sync_report.md`.
5. `persistence_status` не может быть финальным без перечитывания, связки реестра/состояния/события и проверочных доказательств.
6. Если состояние противоречит команде пользователя или целевой путь принадлежит другому режиму, агент создаёт маршрут передачи, фиксирует `blocked` или `partial` и не пишет в чужую область.

## Ошибки состояния

| Ошибка | Действие |
|---|---|
| `active_mode` не совпадает с целевым путём | остановить запись, маршрутизировать через `mode_routing.md` |
| `active_issue` отсутствует для задачи с записью | создать или восстановить задачу через `issue_lifecycle.md` |
| `loaded_context_scope` не покрывает цель | дочитать контекст до записи |
| `current_stage` и marker `stage` расходятся | обновить состояние и маркер, не объявлять финал |
| `next_step` ведёт мимо зависимости | зафиксировать блокировку и якорь возврата |

## Проверочные заметки P2-003

P2-003 фиксирует каноническую связь между `state`, `mode`, `context_loading` и маркером ответа. Полная финальная приёмка остаётся за P2-010: этот протокол задаёт контракт, а не объявляет весь репозиторий готовым.
