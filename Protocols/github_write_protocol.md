# Протокол записи в `GitHub`

[← К реестру протоколов](protocol_index.md)

## Цель

Сохранять production-файлы проверяемо: сначала причина и scope, потом запись, затем перечитывание.

## Preflight

Перед записью агент фиксирует:

- `active_mode` и `active_object`;
- `active_issue` или утверждённую команду пользователя;
- `allowed_write_scope`;
- список создаваемых и изменяемых файлов;
- почему каждый файл является production-файлом;
- какие registry, state и event-файлы меняются вместе с ним;
- план перечитывания и проверки.

## Production-only filter

В production-репозиторий не переносятся:

- архив задания;
- стартовое сообщение агента-исполнителя;
- methodology/roadmap реализации как рабочие документы;
- checkpoint-архивы;
- промежуточные отчёты реализации;
- временные черновики без production-владельца.

## Запись

1. Записать только файлы из разрешённого scope.
2. При конфликте остановить переход и подготовить merge-plan.
3. После записи перечитать изменённые файлы из `GitHub`.
4. Сверить ключевые признаки: заголовок, ссылки, state/status, registry entry.
5. Обновить `persistence_status`: `synced`, `partial`, `failed` или `conflict`.
6. Дать sync-report: путь, действие, commit/status, результат проверки.

## Восстановление согласованности

Если запись нарушила навигацию, state, язык, registry или режимные границы, агент сначала восстанавливает согласованное состояние, фиксирует событие в `Issues/issue_events.jsonl` и только потом продолжает.
